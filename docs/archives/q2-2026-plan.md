# Q2 2026 Plan: Pega Inference Stack

**Status**: Archived
**TL;DR**: Archived Q2 2026 planning snapshot. It is preserved as period planning context for batching hardening, PegaFlow integration, and differentiation work, not as the current active roadmap.

---

## Context

PegaInfer is at single-request parity with vLLM 0.18 on RTX 5070 Ti (Qwen3-4B TPOT 11ms, Qwen3.5-4B TPOT 11.8ms). Continuous batching landed (commit 24584a0): 4-concurrent decode shows 3.65x throughput scaling with near-flat TPOT.

PegaFlow (v0.17) is a production-grade KV cache engine with RDMA, SSD offloading, and prefix caching — currently integrated with vLLM/SGLang via Python PyO3 connectors.

The Q2 thesis: **PegaInfer + PegaFlow = a fully Rust-native inference stack that eliminates the Python GIL bottleneck end-to-end.** This is not "vLLM rewritten in Rust" — it is a fundamentally different architecture that enables capabilities Python frameworks cannot deliver.

## Workstream 1: PegaInfer Hardening (weeks 1-5)

Foundation work. Nothing else matters until batching is stable under load.

### 1.1 Paged KV for Qwen3.5

Qwen3 paged KV is done. Qwen3.5 still uses contiguous KV cache. Prerequisite: FlashInfer HD256 kernel variant for Qwen3.5's 256-dim linear attention layers.

**Exit criteria**: Qwen3.5-4B e2e test passing with paged KV, no accuracy regression.

### 1.2 Concurrent stress testing

Current data: 4-concurrent, 3.65x speedup. Need 16/32/64 concurrent p99 curves to find the throughput ceiling and identify bottlenecks (scheduler latency? CUDA Graph switching? KV allocator fragmentation?).

Must include prefix caching scenarios — vLLM has 147x TTFT head-of-line blocking under asymmetric prefix batches (#37308). If we can show flat TTFT under the same workload, that is a concrete competitive data point.

**Exit criteria**: 64-concurrent p99 TPOT < 15ms, TTFT p99 < 500ms (Qwen3-4B, single GPU).

### 1.3 Chunked prefill (low priority)

Prefill-priority scheduling causes TTFT to grow linearly with concurrency (serial prefill). Chunked prefill interleaving is necessary for acceptable TTFT at high concurrency — without it, the 10th request waits for 9 full prefills.

**Why low priority**: Unified forward pass already folds decode tokens into prefill GEMMs for free. The ITL p99 gap (291ms vs vLLM 211ms) only manifests under sustained high concurrency with large prompts — not the near-term target workload. Qwen3.5 paged KV and stress testing have higher ROI.

**Exit criteria**: TTFT p99 sublinear growth at 16+ concurrency.

## Workstream 2: Pega x Pega (weeks 4-8, overlaps W1)

This is the moat. PegaFlow's Rust core can be called in-process from PegaInfer — no gRPC serialization, no Python FFI, no GIL.

### 2.1 In-process PegaFlow integration

Replace gRPC with direct Rust API calls. PegaInfer's paged KV allocator hands block handles to PegaFlow's StorageEngine for offload/prefetch.

**Key design decision**: PegaFlow as a library crate (`pegaflow-core`) linked into PegaInfer, or as a co-located process with shared-memory IPC? Library crate is simpler and faster; separate process is more flexible for multi-tenant deployment. Start with library crate, keep the interface clean enough to split later.

### 2.2 Prefix caching

PegaFlow already does content-hash block dedup with LRU/TinyLFU admission. Wire this into PegaInfer's tokenizer → KV path so repeated prefixes skip prefill entirely.

**Exit criteria**: Demo showing warm-start TTFT < 1/5 of cold-start TTFT on a repeated-prefix workload.

### 2.3 CPU offloading for long context

When KV cache exceeds GPU VRAM, spill pages to PegaFlow's pinned memory pool (30GB default). This extends effective context length without OOM.

**Exit criteria**: 32K+ context generation that would OOM without offloading completes successfully, with < 20% TPOT degradation.

## Workstream 3: Differentiation (weeks 6-12, pick by ROI)

Each item is independently shippable and blog-worthy.

### 3.1 Observability — the product angle

Observability in LLM serving today is bolted-on and shallow. vLLM has basic Prometheus. SGLang's tracing roadmap (#13511) is half-implemented, their Prometheus has unbounded label cardinality causing memory leaks (#20814), and cache hit rate metrics break under MTP (#20451). Neither has end-to-end request tracing.

This is not just a feature gap — it is a **product positioning opportunity**. The users who care about observability are production teams, and production teams are the ones who pay. What they actually need:

**Request-level tracing (the "what happened to my request" question)**:
- Full lifecycle: queue wait → tokenize → prefill (per-chunk if chunked) → decode steps → detokenize → SSE flush
- Each span carries: batch size at that step, KV cache pages allocated/hit, tokens generated, sampling params
- OpenTelemetry-native so it plugs into existing Jaeger/Tempo/Datadog pipelines without glue code

**Capacity planning metrics (the "can I serve 10x more traffic" question)**:
- KV cache utilization (pages allocated / total pages) as a time series, not a snapshot
- Batch size histogram over time — reveals whether the scheduler is actually batching or running bs=1
- Queue depth + wait time distribution — the leading indicator before latency degrades
- Prefill vs decode GPU time ratio — tells you when chunked prefill is actually needed

**Anomaly detection (the "why did p99 spike at 3am" question)**:
- Per-request TTFT/TPOT exported as histogram, not just averages
- Scheduler decision log: why was this request delayed? (KV full? batch at cap? prefill in progress?)
- Structured logs with request ID correlation — grep one ID, see everything

**Why Rust makes this different from vLLM/SGLang**:
- Tracing instrumentation (tokio-tracing spans) has near-zero overhead when the subscriber is disabled — no "tracing tax" in production
- No GIL means metric collection on a background thread never blocks the scheduler or forward pass
- PegaFlow already ships Prometheus + OTLP — same stack, unified dashboards

**Shippable increments**:
1. Prometheus endpoint with the capacity planning metrics (1 week) — immediately useful, blog-worthy
2. Per-request tracing with OTLP export (1 week) — the differentiator demo
3. Grafana dashboard template in the repo (days) — lower the adoption barrier

### 3.2 Linear attention inference optimization

Qwen3.5's hybrid architecture (24 linear + 8 full attention) is unique to pegainfer. FLA has minimal inference investment. Opportunities: fused recurrent kernels, state compression, speculative decoding with linear-attention drafters.

See `nonstandard-attention-milestone.md` for the broader direction.

### 3.3 P/D disaggregation

PegaFlow's RDMA layer (sideway crate, RoCE/IB) is production-tested. Split PegaInfer into prefill nodes and decode nodes, with PegaFlow handling KV transfer between them.

### 3.4 Parallel tokenization and hashing

Without GIL, tokenizer encode/decode and prefix hash computation can run on dedicated threads, fully parallel with GPU work. vLLM/SGLang cannot do this — their tokenizer calls hold the GIL.

## Future: MTP / Speculative Decoding (not this Q)

MTP (Multi-Token Prediction) and speculative decoding are on the radar but explicitly deferred to Q3. Reasons to defer: W1 batching must be rock-solid first, and the competitive landscape shows this feature is still buggy even in mature frameworks.

Context for when we pick this up:
- vLLM's MTP causes **76% latency regression** on Qwen3-Next-80B (#35387), suspected CPU-GPU sync in `mamba_postprocess`
- vLLM's speculative decode **crashes on hybrid models** (#38196) — GDN backend asserts mutual exclusivity between decode types
- SGLang's Eagle+NGram hybrid has **illegal memory access** (#9481), long-standing and hard to reproduce
- SGLang's MTP **breaks prefix cache hit rate** (#20451)

Our advantage when we do build this: Rust scheduler can overlap draft/verify without GIL contention, and our native RecurrentState means linear-attention layers don't need special-casing in the speculative decode path. A linear-attention drafter model is an interesting research direction — fast draft with O(1) state, verify with full attention.

## Go-to-Market

### Narrative

**Not** "vLLM rewritten in Rust." **Instead**: "What inference looks like without the GIL."

Core claims, each backed by data:
- **Hybrid attention native**: Qwen3.5 works correctly and efficiently out of the box, while vLLM/SGLang are still patching crashes
- **p99 stability**: no GC pauses, no GIL contention → flat tail latency under sustained load
- **Instant startup**: build-time kernel compilation (nvcc + Triton AOT), no torch.compile, no Inductor — seconds vs vLLM's ~215s for 70B models
- **Native observability**: per-request tracing without blocking inference
- **Native distributed**: RDMA KV transfer without C++ sidecar or Python FFI

### Startup time as a marketing point

vLLM Llama-70B TP=8 startup breakdown (from issue #19824):
- P2P connectivity check: ~60s
- Weight loading: ~10s
- Dynamo tracing: ~15s
- torch.compile: ~70s
- CUDA Graph capture: ~60s
- **Total: ~215 seconds**

Users are forced to use `--enforce-eager` (skip torch.compile + CUDA graph) to get reasonable startup, sacrificing runtime performance. This is architecturally unfixable — it comes from depending on PyTorch's compilation stack.

PegaInfer compiles all CUDA/Triton kernels at build time. Runtime startup is: load weights → allocate buffers → capture CUDA graph. No JIT, no dynamo, no Inductor. Measure and publish this number — it will be a compelling data point for autoscaling and development iteration use cases.

### Current landscape snapshot (2026-03-29)

**vLLM (v0.18.0, 2026-03-20)**:
- Shipping aggressively: 5 releases in 8 weeks, 213 contributors per release
- V1→V2 model runner transition is bumpy — V2 crashes on Qwen3.5 (#38041)
- Hybrid attention is their Achilles' heel: KV cache 7x overestimation (#37121), prefix caching < 0.1% hit rate (#36493), spec decode crashes (#38196), V1 deadlocks under fp8 + prefix + Qwen3.5 (#37729)
- Python overhead explicitly acknowledged in V1 RFC (#8779): "Python is slow, GPUs will get faster, CPUs will not"
- Startup time ~215s for 70B TP=8 (#19824)
- Prefix caching: 147x TTFT HOL blocking under asymmetric batches (#37308)
- torch.compile as default compilation backend — deep PyTorch coupling
- Focus areas: speculative decoding (buggy on hybrid), disaggregation (NIXL-EP), multi-hardware (spreading thin across ROCm/XPU/CPU/Metal)

**SGLang (v0.5.10rc0, 2026-03-28)**:
- Monthly release cadence, model-first strategy (day-0 support for hot models, fix bugs after)
- Qwen3.5 is a bugfest: 2 tracking issues (#18590, #20069), 15+ open bugs spanning PD disagg, TP/PP, quantization, CPU offload, determinism
- GDN kernel accuracy bug (#20791): FlashInfer's decode kernel gives wrong results under `no_buffer` scheduling — state aliasing when SSM slots are reused
- Observability is a known gap: tracing half-implemented (#13511), Prometheus label cardinality causes memory leak/DoS (#20814), cache hit metrics broken under MTP (#20451)
- "Zero-overhead scheduler" claim disputed by users (#19347) — CPU scheduling may not actually overlap GPU
- Unified radix cache refactor (#20415) plans Stage 3 "Rust rewrite" — acknowledged as necessary but distant
- Focus areas: DeepSeek V3.2/GLM-5 optimization, SM120/Blackwell, diffusion serving, RL integration, multi-modal (SGLang-Omni)

**Shared weaknesses**:
- Both architectures assume uniform attention layers — Qwen3.5's hybrid design breaks this assumption across KV allocators, CUDA graph dispatch, prefix caching, and speculative decoding
- Neither has a path to remove Python from the serving hot path — vLLM optimizes within Python, SGLang aspires to Rust but hasn't started
- Observability is bolted-on in both, not designed-in

## Risks

1. **Premature jump to W2/W3 before W1 is solid.** Continuous batching p99 data is the foundation of every downstream claim. If it's not stable, the "no GIL = stable p99" narrative falls apart.
2. **PegaFlow integration scope creep.** Start with the simplest integration (library crate, prefix caching only), prove value, then expand.
3. **Blog timing vs. readiness.** Don't publish benchmarks that can't be reproduced. Every number in a blog must come from a committed, runnable benchmark script.
4. **Competitive response.** vLLM has 213 contributors per release. They can throw bodies at hybrid attention bugs. Our advantage is architectural, not headcount — maintain that by staying focused, not by trying to match feature breadth.
