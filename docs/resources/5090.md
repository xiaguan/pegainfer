## Build

Always use release mode and set the 5090 SM target explicitly.

```bash
ssh 5090 'bash -ic "cd /root/develop/xingming/pegainfer && PEGAINFER_CUDA_SM=120 cargo build --release"'
```

This completed successfully after submodules were initialized.

Current build-target audit:

- `PEGAINFER_CUDA_SM=120` makes `crates/pegainfer-kernels/build.rs` compile native CUDA objects for `sm_120` and embed a `compute_120` PTX fallback.
- `cuobjdump -lelf` on both `libkernels_cuda.a` and `target/release/pegainfer` shows `sm_120.cubin`, so release runs are not relying on PTX JIT for Qwen3 kernels.
- CUDA `13.1` can compile the same Qwen3 profiling binary into `sm_120` cubins, but running it on the current box fails with `cudaError=35` because it links `libcudart.so.13` and the installed driver reports CUDA `12.9` support. Use CUDA `12.9` until the driver is upgraded.

## Frontend Boundary

The local process starts `vllm-server`; `vllm-text` owns tokenizer loading, prompt encoding, streaming detokenization, chat templates, and reasoning parsing. pegainfer owns model loading, scheduling, CUDA execution, sampling, and token-id output over the local engine-core bridge.

Do not reintroduce a pegainfer tokenizer wrapper or local OpenAI HTTP route layer unless `vllm-frontend-rs` cannot provide the behavior.
